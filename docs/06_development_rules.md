# 06. Development Rules

## Источник правды

- Один источник правды для кода — папка `src/`.
- Изменения кода делает Codex в `src/`.
- Roblox Studio не редактируем вручную для кода, сервисов и клиентских скриптов.
- Roblox Studio можно использовать для карты, `Workspace`, `Baseplate`, `SpawnLocation` и визуальной проверки.

## Rojo и структура проекта

- Rojo должен быть подключён во время разработки и тестирования кода.
- Roblox Studio получает код из `src` через Rojo.
- `Workspace` не мапить через Rojo.
- `default.project.json` не трогать без необходимости.
- Не допускать дублей `ServerMain`, папки `Services`, `StarterPlayerScripts` и других синхронизируемых контейнеров.

## Проверки после каждого этапа

- Проверить, что в Roblox Studio нет дублей серверных скриптов, сервисов и клиентских скриптов.
- Проверить, что нет Git conflict markers. Не оставлять в файлах строки маркеров начала, разделителя и конца конфликта.
- Проверить, что квестовый цикл не сломан: староста, сбор 3 деревьев, сдача квеста, открытие участка.
- Проверить, что UI ресурсов и UI квеста сосуществуют и не мешают друг другу.
- Проверить, что DataStore-ошибки в Studio не ломают запуск игры.

## Git и артефакты

- После успешного теста делать git commit.
- `.rbxl.lock` не коммитить.
- Не коммитить временные файлы Roblox Studio.
- Не трогать `.rbxl` без явной необходимости и понимания, что изменилось в Studio.

## Практическое правило

Если нужно изменить игровую логику, сервис, UI или клиентский код — править `src`. Если нужно изменить карту, расположение базовых объектов мира или ручные объекты `Workspace` — править в Roblox Studio, не через Rojo-маппинг `Workspace`.
