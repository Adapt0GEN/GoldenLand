# 06. Architecture Migration — Personal Islands and Shared Locations

## Назначение документа

Этот документ фиксирует план перехода GoldenLand от текущей компактной MVP-архитектуры к будущей архитектуре с личными островами игроков и общими локациями.

Документ не описывает немедленную реализацию всех систем. Его цель — задать направление, чтобы дальнейшие задачи для Claude/Codex не усиливали временные решения, а постепенно готовили проект к правильной архитектуре.

## Текущая MVP-архитектура

Сейчас проект работает как компактный одиночный вертикальный срез.

В одной быстро тестируемой зоне находятся:

- стартовая площадка игрока;
- NPC Староста;
- стартовые ресурсы;
- личная земля игрока;
- дом;
- склад;
- мастерская;
- кузница;
- ForestZone;
- RockZone.

Это удобно для проверки раннего игрового цикла, потому что все важные объекты находятся рядом и быстро доступны во время Play-тестов.

Текущие основные сервисы:

- PlayerDataService — профиль игрока, загрузка, сохранение, публичные данные для UI.
- WorldService — стартовый мир, WorldRoot, проходы, ForestZone, RockZone, визуальное состояние зон.
- ResourceService — стартовые ресурсы, ресурсы ForestZone и RockZone.
- PlotService — личный участок, дом, склад, мастерская, кузница, улучшения, визуалы зданий, prompts и часть производственной логики.
- CurrencyService — выдача, списание и отображение ресурсов.
- QuestService — квестовая логика.
- NPCService — NPC и взаимодействия.
- ClientMain — UI ресурсов, квестов, сообщений и Action Preview.

## MVP-only ограничения

Текущая архитектура подходит для раннего одиночного MVP, но не подходит как финальная основа для игры с личными островами и общими локациями.

Главные ограничения:

1. Личная земля пока не является полноценным личным островом.

Сейчас участок игрока создаётся рядом со стартовой зоной в компактном MVP-layout. Это удобно для тестирования, но в будущем личная территория игрока должна быть изолированным островом или отдельным персональным контейнером.

2. ForestZone и RockZone сейчас являются глобальными объектами мира.

Если WorldService создаёт одну глобальную ForestZone или RockZone, то прогресс одного игрока может влиять на визуальное состояние мира для других игроков.

Например, если один игрок открыл ForestZone и глобальный завал исчез, другой игрок может увидеть мир уже изменённым, хотя в его профиле зона ещё не открыта.

3. Персональный прогресс игрока сейчас может менять глобальные объекты.

Это главный архитектурный риск. Любое состояние, зависящее от профиля игрока, не должно удалять, создавать или менять общие объекты мира для всех игроков.

4. PlotService стал слишком широким сервисом.

Сейчас PlotService одновременно отвечает за:

- создание участка;
- восстановление участка;
- визуалы дома;
- визуалы склада;
- визуалы мастерской;
- визуалы кузницы;
- строительство;
- улучшения;
- forge production loop;
- prompts;
- часть сообщений игроку.

Для раннего MVP это допустимо, но дальше сервис нужно будет постепенно разделять.

5. Стартовые ресурсы и персональные ресурсы пока не разделены явно.

Сейчас ResourceService создаёт стартовые ресурсные точки глобально. В будущем нужно различать:

- общие ресурсы стартового мира;
- ресурсы личного острова;
- ресурсы общих локаций;
- ресурсы экспедиционных/гильдейских зон.

## Целевая архитектура

Будущая архитектура должна разделять три слоя мира:

1. SharedWorld

Общий стартовый мир или общий хаб.

Возможное содержимое:

- стартовая зона;
- Староста;
- обучающие NPC;
- входы в общие локации;
- вход на личный остров;
- базовые общие ориентиры;
- будущий общий город, если он останется в том же place.

SharedWorld не должен зависеть от персонального прогресса одного игрока.

2. PlayerIslands

Контейнер личных островов игроков.

Каждый игрок должен иметь свой собственный остров или персональный контейнер:

    PlayerIslands
    └─ Island_<UserId>

Внутри острова должны жить все объекты, зависящие от личного прогресса игрока:

- личная земля;
- дом;
- склад;
- мастерская;
- кузница;
- личные зоны развития;
- личная ForestZone;
- личная RockZone;
- персональные ресурсные точки;
- персональные заблокированные проходы;
- визуальное состояние освоения острова.

3. SharedLocations

Общие локации для всех игроков.

Возможное содержимое:

- общий город;
- рынок;
- гильдейская зона;
- экспедиционные ворота;
- рейдовые зоны;
- будущие нейтральные земли;
- общие PvE/PvP-карты.

SharedLocations должны управляться серверным или общим состоянием, а не профилем одного игрока.

## Предлагаемая будущая структура Workspace

Целевая структура может выглядеть так:

    Workspace
    ├─ WorldRoot
    ├─ PlayerIslands
    │  └─ Island_<UserId>
    │     ├─ IslandBase
    │     ├─ Buildings
    │     ├─ Zones
    │     ├─ Resources
    │     └─ Markers
    └─ SharedLocations
       ├─ CommonTown
       └─ ExpeditionGates

Пояснения:

- WorldRoot может временно остаться для совместимости с текущим MVP.
- PlayerIslands должен стать основным корнем персональной прогрессии.
- SharedLocations должен использоваться для будущих общих зон.
- Island_<UserId> должен быть изолированным контейнером прогресса конкретного игрока.
- Buildings должны содержать личные здания игрока.
- Zones должны содержать личные зоны острова: ForestZone, RockZone и будущие зоны.
- Resources должны содержать персональные ресурсные точки острова.
- Markers должны содержать указатели, таблички, входы, декоративные маркеры и временные визуальные элементы.

## Главное архитектурное правило

Любое состояние, зависящее от профиля игрока, должно изменять только личный контейнер этого игрока.

    Player profile state must mutate only that player's island/container, not global shared world objects.

Общие локации должны управляться общим серверным состоянием, а не профилем одного игрока.

    Shared locations must be driven by server-wide/shared state, not by one player's personal profile.

## Что должно стать персональным

Следующие системы должны постепенно переехать в личный остров игрока:

- личный участок;
- дом;
- склад;
- мастерская;
- кузница;
- личная ForestZone;
- личная RockZone;
- личные blocked paths;
- личные resource zones;
- визуальное состояние освоения острова;
- состояние очищенных зон;
- персональные resource prompts.

## Что может остаться общим

Следующие системы могут оставаться в общем мире:

- стартовая зона;
- Староста;
- обучающие NPC;
- общие указатели;
- вход на личный остров;
- входы в общие локации;
- будущий общий город;
- рынок;
- социальные зоны;
- экспедиционные ворота.

## Миграционный план

### Шаг 1. Зафиксировать этот архитектурный документ

Создать этот документ и использовать его как ориентир для следующих задач.

Цель шага — не менять код, а зафиксировать направление.

### Шаг 2. Добавить безопасные контейнеры мира

Добавить создание контейнеров:

- PlayerIslands;
- SharedLocations.

На этом шаге нельзя менять игровой процесс. Нужно только подготовить структуру Workspace и helper-функции.

### Шаг 3. Ввести PersonalIslandService

Создать новый сервис:

    src/ServerScriptService/Services/PersonalIslandService.lua

Ответственность сервиса:

- найти или создать Island_<UserId>;
- создать IslandBase;
- назначить OwnerUserId и OwnerName;
- вернуть islandModel и islandBase;
- не заниматься зданиями;
- не заниматься ресурсами;
- не заниматься квестами;
- не заниматься производством.

PlotService.CreateTestPlot(player) должен временно остаться как compatibility wrapper, чтобы не ломать существующие вызовы.

### Шаг 4. Перенести создание личного участка в PlayerIslands

Перенести текущую генерацию личного участка из компактной MVP-позиции в:

    Workspace.PlayerIslands.Island_<UserId>

При этом важно сохранить:

- восстановление дома;
- восстановление склада;
- восстановление мастерской;
- восстановление кузницы;
- текущие prompt-действия;
- UI;
- сохранение прогресса.

### Шаг 5. Перенести ForestZone и RockZone под личный остров

ForestZone и RockZone должны стать персональными зонами:

    Island_<UserId>
    └─ Zones
       ├─ ForestZone
       └─ RockZone

После этого:

- ForestUnlocked должен влиять только на ForestZone конкретного игрока;
- RockZoneUnlocked должен влиять только на RockZone конкретного игрока;
- очистка леса не должна менять глобальный мир;
- открытие RockZone не должно менять глобальный мир;
- визуальное состояние зон должно восстанавливаться внутри острова игрока.

### Шаг 6. Разделить PlotService

PlotService нужно постепенно разгрузить.

Целевое разделение:

1. PersonalIslandService

Отвечает за:

- контейнер острова;
- позиционирование острова;
- IslandBase;
- базовые атрибуты владельца.

2. BuildingService

Отвечает за:

- уровни зданий;
- проверки возможности строительства;
- проверки возможности улучшения;
- применение уровней зданий;
- стоимость строительства и улучшений.

3. BuildingVisualService

Отвечает за:

- временные Part-модели зданий;
- визуальное восстановление зданий;
- таблички;
- build spots;
- prompt parts.

4. ProductionService

Отвечает за:

- рецепты кузницы;
- плавку Metal в MetalIngot;
- производство MetalParts;
- будущие производственные цепочки.

5. PlotService

Временно остаётся как слой совместимости.

На позднем этапе PlotService можно будет либо удалить, либо оставить как фасад над PersonalIslandService и BuildingService.

### Шаг 7. Добавить первую общую локацию

Только после стабилизации личных островов можно добавлять первую общую локацию.

Первый безопасный вариант:

    SharedLocations
    └─ CommonTown

На старте это может быть простая зона с табличкой, NPC и входами в будущие активности.

Не нужно сразу добавлять TeleportService. Для MVP общая локация может существовать в том же Roblox place.

## Когда понадобится TeleportService

TeleportService не нужен на ближайших шагах.

Он понадобится позже, если:

- личные острова станут слишком большими;
- общий город будет отдельным place;
- появятся отдельные рейдовые карты;
- появятся гильдейские войны;
- появятся отдельные PvP/PvE зоны;
- серверу станет тяжело держать всё в одном place.

До этого безопаснее развивать архитектуру внутри одного place.

## Non-goals

Этот миграционный план не реализует:

- боёвку;
- классы;
- инвентарь;
- рюкзак;
- еду;
- усталость;
- питомцев;
- автоматизацию;
- рабочих;
- рейды;
- гильдейские войны;
- TeleportService;
- большую процедурную генерацию карты;
- полный рефакторинг всех сервисов за один шаг.

## Правило маленьких шагов

Миграция должна выполняться маленькими проверяемыми задачами.

Каждый шаг должен:

- сохранять текущий рабочий игровой цикл;
- не ломать стартовый квест;
- не ломать добычу ресурсов;
- не ломать UI;
- не ломать сохранение;
- не ломать дом, склад, мастерскую и кузницу;
- не менять default.project.json без явной необходимости;
- не создавать src/Workspace;
- не трогать R15/R6/avatar settings.

## Ближайшая рекомендуемая задача после этого документа

Следующая безопасная кодовая задача:

    Add world container helpers for PlayerIslands and SharedLocations without changing gameplay.

Цель следующей задачи:

- создать контейнер Workspace.PlayerIslands;
- создать контейнер Workspace.SharedLocations;
- не переносить туда текущие объекты;
- не менять игровой процесс;
- не менять координаты текущего MVP;
- подготовить основу для PersonalIslandService.
